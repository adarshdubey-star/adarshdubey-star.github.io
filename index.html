ChromaDB Codebase Index

Poke splits our entire codebase into 1032 code chunks — one per function or class — and indexes them by meaning using embeddings.

When reviewing a file, it searches this index for the 5 most related chunks from other files.

Why this matters

If a parameter is removed from host.py, Poke pulls up the function in serialization.py where that logic moved to. The LLM sees both sides — so it doesn't flag a refactor as a bug.

This is why an 18-file refactoring PR gets 0 false positives instead of 36.

Persists locally. Rebuilds only when files change.

How Poke Thinks — AI + Quality

Two models working together, plus smart deduplication

Two-Model Strategy

GPT-4o Review

Generates findings. For each file, it sees:

  • The diff — what changed
  • The complete file at HEAD — not just the diff
  • Related code from other files via ChromaDB
  • The PR description — author's intent
  • Which other files changed in this PR
  • Our team rules + auto-discovered facts

Each finding gets a confidence score: 1-10. Below 6 = auto-dropped.

o3-mini Critique

A reasoning model that validates every finding from GPT-4o.

It asks: "Is this a real issue, or did the code just move to another function? Did behavior actually change, or is this just refactoring?"

Only findings that pass this check survive. This is where most false positives die.

Both models are free on GitHub Models — zero cost.

Smart Deduplication

If 14 files all have the same org_id concern, you don't want 14 comments saying the same thing.

Step 1: Concept Grouping

Recognizes domain patterns in findings. If multiple findings mention "org_id" + "cross-tenant", they're about the same thing.

14 findings → 1 finding that says "org_id concern — also affects these 13 files"

Step 2: Keyword Similarity

For findings that don't match a known pattern, compares the actual words. If two findings share more than 40% of their meaningful words, they get merged.

Step 3: Per-Lens Cap

Maximum 5 findings per lens, highest severity first. No single lens can flood the review.

Result: focused, actionable reviews — not walls of noise.
